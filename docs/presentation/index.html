the problem

Engineering runs on toil.

  • Answering “how does this part work?” for the hundredth time.
  • Chasing dependency updates until something falls behind and bites.
  • Docs that quietly drift away from what the code actually does.
  • Re-triaging the same kinds of problems across systems.

What if a small fleet of AI specialists handled the toil — and you stayed in control?

the idea

One shared runtime.
Many focused agents.
Each does one job.

Not autonomous

Every agent only reads, advises, or proposes. A person approves anything that goes live.

One foundation

The shared runtime does the mechanical work in plain, deterministic code — no AI for the plumbing. Predictable, repeatable, and cheaper: the coding agent runs only where judgment is needed.

Cloud-native

Hosted as managed cloud services (AWS) on a shared catalog of your projects — the same access controls, monitoring, and reliability as the rest of your stack.

Every step ends with a recommendation, not an action — each agent hands back to a person, and validate-it just comments whether it worked. Humans move the ticket. Always.

Rollout: groom-it is live; the other steps are implemented and running dark, flipped on one at a time as each earns trust.

trust by design

Built to be safe to say yes to.

  • Human in the loop — nothing consequential without a person.
  • Proposes, never merges — write-capable agents only open changes.
  • Opt-in — an agent touches a project only when enabled for it.
  • Least privilege — each agent can reach only what its job needs.
  • Data stays isolated — the AI never holds database credentials.
  • Cross-model review — a different AI critiques the code’s author (rolling out).
  • Auditable — actions logged; ticket/doc access is scoped.
  • Cost-tracked — know what each ticket cost in time & tokens (planned).

why it’s fun

The clever bits.

No central brain

Agents never call each other — they meet at the ticket and the PR. Your existing workflow is the orchestrator.

Two models argue

Review can run as a Claude-vs-Codex debate to agreement (when the second runner is configured) — a genuinely independent second opinion on the code.

Compose, don’t bloat

One runtime; agents are thin. New behavior is usually just a new skill — small, focused, shared.

Pay only for judgment

Deterministic plumbing plus the gate mean the coding agent runs only when real judgment is needed — not on the mechanical or irrelevant parts.
